## 计算机网络

osi 七层协议是哪七层？

- 物理、数据链路、传输、会话、表示、应用

DNS 端口号？

- 53
- 其他端口：FTP 21, SMTP 25, HTTP 80, HTTPS 443

访问网站发生了什么？

- 向 DNS 服务器查询 IP
- 向 web 服务器发送 HTTP 请求
- 服务器发送网页内容

应用层有哪些协议？

- HTTP, DNS, FTP, SMTP, POP3, P2P（不用全面）

http 有哪些字段？

- 请求：
  - Accept
  - User-Agent
  - Connection
  - Cookie
- 响应：
  - Content-Length
  - Content-Type
  - Date
  - Set-Cookie

http 状态码有哪些，对应英文？

- 200 OK 请求成功
- 301, 302, 303 重定向
- 304 Not Modified 未修改
- 400 Bad Request 语法错误
- 403 Forbidden 拒绝请求
- 404 Not Found 资源不存在
- 500 Internal Server Error 服务器内部错误
- 502 Bad Gateway 网关或代理得到了无效响应

tcp 和 udp 的区别？

- tcp 发送数据前需要连接
- tcp 提供可靠服务
- tcp 传输开销大，占用系统资源多
- tcp 必须是点到点的，udp 没有限制

- ARP, IP, ICMP, IGMP, RIP, OSPF 等

数据链路层通过什么来转发？

- MAC 地址

### TCP

三次握手

- SYN >, SYN ACK <, ACK >
- 之后双方进入 Established 状态
- 为什么不是两次：两次握手接收方无法得知第二次握手的消息是否送达

四次挥手

- FIN >, ACK <, FIN <, ACK >
- 第四次挥手后，发送方进入 TIME_WAIT 状态，等待两个最大段生命周期 MSL，可能有一分钟，为了收到可能的第三次挥手的重传
- 为什么需要四次：发送方的内容发送完后会进行前两次挥手，此后接收方的内容可以继续发送

超时重传 RTO

- 往返时间 RTT：SYN 发出到 ACK 接受之间的时间
- SRTT：平滑的 RTT
- RTTVAR：SRTT 与真实值的差距
- RTO：SRTT + 4 RTTVAR
- 超过 RTO 会重传

快速重传

- 多个消息发送后，其中有一个 seq=x 丢失，接收方会接收后面的包，每个包回复 ack=x
- 发送方收到三个 ack=x（不包括第一个正确的 ACK）后重发 seq=x 的包
- SACK 可以在 ACK 包里写收到的序列和范围

流量控制

- 发送端和接收端都有一个滑动窗口
- 头部字段 16 位表示滑动窗口的大小

拥塞控制

- 拥塞窗口 cwnd
- 慢启动阀值 ssthresh，一开始是 65536 字节
- 慢启动：每个 ACK cwnd++，指数上升，直到 cwnd > ssthresh
- 拥塞避免：每过一个 RTT cwnd++，线性上升
- RTO 超时重传：应该是一个 RTO 内没有回复，ssthresh = cwnd / 2，cwnd = 1，回到慢启动阶段
- 快速重传：3 个重复 ACK，cwnd = cwnd / 2，ssthresh = cwnd，进入快速恢复
- 快速恢复：cwnd += 3，之后每个重复的 ACK 都加一

半连接队列、SYN Flood 攻击

- 半连接队列：服务器收到第一次握手后，将这个连接推入半连接队列
- 全连接队列：三次握手结束后，将这个连接推入全连接队列
- SYN Flood 攻击：伪造不存在的 IP 向服务器发起大量 SYN 报文
- syn cookie：将 cookie 值作为序列号发送 SYNACK 包，ACK 包接收后，若 cookie 值正确再建立连接
- SYN Proxy：用防火墙代理

### HTTP

HTTPS

- 在 TCP 和 HTTP 之间加入了 SSL/TLS 安全协议

HTTPS 连接过程

- 首先是 TCP 三次握手，然后是下面的 SSL 四次握手
- ClientHello >
  - 客户端支持的 SSL/TLS 协议版本
  - 客户端随机数
  - 密码套件列表（哪些算法）
- ServerHello <
  - 确认的 SSL/TLS 协议版本
  - 服务器随机数
  - 确认的密码套件列表
  - 数字证书（包含证书机构、所有者、签名、公钥）
- 客户端回应 >
  - 用公钥加密的随机数，作为会话密钥（对称加密）
  - 加密算法改变通知
  - 校验
- 服务器回应 <
  - 加密算法改变通知
  - 校验

HTTP1.1

- 支持长连接，Connection: Keep-Alive，但是完成一个请求-响应后才能进行下一个（但是有说可以并发连接 2-8 个）

HTTP2

- 头部压缩为二进制
- 多路复用
  - 将一个连接分为多个流，多个流复用一条 TCP 连接
  - 一个流可以发多个消息，每个消息分割为更小的帧，并编号（流 ID）
  - 同一个流必须有序发送帧，不同流可以并发发送
- 支持全双工

HTTP3

- 将传输层协议改为 UDP
- 使用 QUIC，将 TCP, SSL 共 7 次握手改为 3 次

## 面向对象

特点：

1. 对象包含数据和行为（有成员变量、成员函数）
2. 封装隐藏了实现细节（内部细节设为私有）
3. 继承作为类型系统和代码共享，也可以实现动态多态
